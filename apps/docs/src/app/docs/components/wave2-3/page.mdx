## Pagination

Page navigation with ellipsis for large page counts.

<ComponentPreview label="Pagination">
  <PaginationDemo />
</ComponentPreview>

```tsx
import { Pagination } from "@weiui/react";

<Pagination page={3} totalPages={20} onPageChange={setPage} />
<Pagination page={1} totalPages={100} onPageChange={setPage} siblingCount={2} />
```

| Prop | Type | Default | Description |
|------|------|---------|-------------|
| page | `number` | — | Current page |
| totalPages | `number` | — | Total page count |
| onPageChange | `(page: number) => void` | — | Page change handler |
| siblingCount | `number` | `1` | Pages shown around current |

### Accessibility

- Root uses `<nav aria-label="Pagination">` so assistive tech announces it as a navigation landmark.
- Current page button carries `aria-current="page"`.
- Previous / Next buttons expose `aria-label` so screen readers hear the action rather than the raw glyph.
- Ellipses are marked `aria-hidden="true"` — they are visual cues only.

## Transfer

Dual-list with items moved between source and target.

<ComponentPreview label="Transfer">
  <TransferDemo />
</ComponentPreview>

```tsx
import { Transfer } from "@weiui/react";

<Transfer
  sourceItems={[
    { value: "react", label: "React" },
    { value: "vue", label: "Vue" },
    { value: "angular", label: "Angular" },
  ]}
  onChange={(source, target) => console.log({ source, target })}
/>
```

### Controlled target values

Drive the selected-side from the outside with `targetValues` + `onTargetValuesChange`. Useful when the selection is part of form state or must be persisted.

```tsx
const [selected, setSelected] = useState<string[]>(["react"]);

<Transfer
  sourceItems={allFrameworks}
  targetValues={selected}
  onTargetValuesChange={setSelected}
/>
```

| Prop | Type | Default | Description |
|------|------|---------|-------------|
| sourceItems | `TransferItem[]` | — | Full set of items available |
| targetItems | `TransferItem[]` | `[]` | Initial items on the target side (uncontrolled only) |
| targetValues | `string[]` | — | Controlled selected values on the target side |
| onTargetValuesChange | `(values: string[]) => void` | — | Fires when the target list changes |
| onChange | `(source, target) => void` | — | Fires with the full source + target lists |
| sourceTitle | `string` | `"Available"` | Heading over the source list |
| targetTitle | `string` | `"Selected"` | Heading over the target list |

### Accessibility

- Both lists use `role="listbox"` with `aria-multiselectable="true"`; individual rows are `role="option"` with `aria-selected` kept in sync with their selection state.
- Each list has an `aria-label` ("Available items" / "Selected items") so screen readers announce which side they're on.
- The whole component is wrapped in `role="group"` with `aria-label="Transfer list"` to expose it as a named landmark.
- Move buttons use `aria-label="Move selected to target"` / `"Move selected to source"` so the direction is unambiguous to screen readers.

## Splitter

Resizable split panes — horizontal or vertical.

<ComponentPreview label="Splitter">
  <SplitterDemo />
</ComponentPreview>

```tsx
import { Splitter } from "@weiui/react";

<Splitter defaultSizes={[60, 40]} style={{ height: "400px" }}>
  <div>Left panel</div>
  <div>Right panel</div>
</Splitter>

<Splitter orientation="vertical" minSize={20}>
  <div>Top</div>
  <div>Bottom</div>
</Splitter>
```

### Controlled sizes

Both panes can be driven from outside via `sizes` + `onSizesChange`. Sizes are an array of percentages that must sum to 100.

```tsx
const [sizes, setSizes] = useState<[number, number]>([30, 70]);

<Splitter sizes={sizes} onSizesChange={setSizes}>
  <Nav />
  <Editor />
</Splitter>
```

### Props

| Prop | Type | Default | Description |
|------|------|---------|-------------|
| orientation | `"horizontal" \| "vertical"` | `"horizontal"` | Axis the panes split along |
| sizes | `[number, number]` | — | Controlled pane sizes as percentages (sum to 100) |
| defaultSizes | `[number, number]` | `[50, 50]` | Initial uncontrolled pane sizes |
| onSizesChange | `(sizes: [number, number]) => void` | — | Fires when the user drags the handle |
| minSize | `number` | `10` | Minimum percentage allowed per pane |

### Accessibility
- Handle uses `role="separator"` with `aria-orientation`, `aria-valuenow`, `aria-valuemin`, and `aria-valuemax`
- Keyboard: Arrow keys resize the panes in 5% steps (Left/Right for horizontal, Up/Down for vertical)
- `minSize` bounds both sides, so the separator can never be dragged past the permitted range

## AppBar

Sticky top navigation header with brand, nav links, and actions.

<ComponentPreview label="AppBar">
  <AppBarDemo />
</ComponentPreview>

```tsx
import { AppBar, AppBarBrand, AppBarNav, AppBarLink, AppBarActions, Button } from "@weiui/react";

<AppBar>
  <AppBarBrand>MyApp</AppBarBrand>
  <AppBarNav aria-label="Primary">
    <AppBarLink active>Dashboard</AppBarLink>
    <AppBarLink>Settings</AppBarLink>
  </AppBarNav>
  <AppBarActions>
    <Button variant="ghost" size="sm">Sign out</Button>
  </AppBarActions>
</AppBar>
```

### Props

`AppBar` and its slot components (`AppBarBrand`, `AppBarNav`, `AppBarActions`) forward every native HTML attribute — there are no WeiUI-specific props on them.

`AppBarLink` adds one:

| Prop | Type | Default | Description |
|------|------|---------|-------------|
| active | `boolean` | `false` | Sets `data-active` and `aria-current="page"` |

### Accessibility

- Root renders as `<header>` — pair it with a page `<main>` landmark so assistive tech distinguishes the banner from body content.
- `AppBarNav` renders as `<nav>`; pass `aria-label="Primary"` so screen readers can distinguish it from other nav landmarks.
- Active links carry `aria-current="page"` so the current view is announced.
- Sticky positioning is CSS-driven — the `.wui-app-bar` class handles it, no prop needed.

## BottomNav

Mobile bottom navigation bar — fixed to viewport bottom. Adds `env(safe-area-inset-bottom)` padding automatically so the bar clears the iOS home indicator.

<ComponentPreview label="BottomNav">
  <BottomNavDemo />
</ComponentPreview>

```tsx
import { BottomNav, BottomNavItem } from "@weiui/react";

<BottomNav>
  <BottomNavItem active icon="🏠" label="Home" onClick={() => navigate("/")} />
  <BottomNavItem icon="🔍" label="Search" onClick={() => navigate("/search")} />
  <BottomNavItem icon="👤" label="Profile" onClick={() => navigate("/profile")} />
</BottomNav>
```

The safe-area inset is applied via CSS on `.wui-bottom-nav` and is therefore zero on platforms without a bottom inset. No consumer-side configuration is needed.

### Props

`BottomNav` forwards all native `<nav>` attributes.

`BottomNavItem`:

| Prop | Type | Default | Description |
|------|------|---------|-------------|
| active | `boolean` | `false` | Marks as current view; sets `aria-current="page"` |
| icon | `ReactNode` | — | Icon rendered above the label |
| label | `string` | — | Visible label text |

### Accessibility

- Root renders as `<nav aria-label="Bottom navigation">` so assistive tech announces it as a named landmark.
- Active items set `aria-current="page"`.
- Icons are marked `aria-hidden="true"` — the `label` prop is the announced text, so screen readers don't speak the decorative glyph.
- Each item inherits the Button 44×44px minimum touch target for reliable thumb reach on mobile.

## SpeedDial

Floating action button that expands into multiple actions.

<ComponentPreview label="SpeedDial">
  <SpeedDialDemo />
</ComponentPreview>

```tsx
import { SpeedDial } from "@weiui/react";

<SpeedDial
  actions={[
    { id: "add", icon: "+", label: "Add item", onClick: () => add() },
    { id: "edit", icon: "✏", label: "Edit", onClick: () => edit() },
    { id: "share", icon: "↗", label: "Share", onClick: () => share() },
  ]}
/>
```

### Keyboard
- Enter / Space on the FAB opens the action list and moves focus to the first action
- ArrowUp / ArrowDown rove focus between actions (wraps at the ends)
- Home / End jumps to the first / last action
- Escape closes the menu and returns focus to the FAB

### Props

| Prop | Type | Default | Description |
|------|------|---------|-------------|
| actions | `SpeedDialAction[]` | — | Action list — each has `id`, `icon`, `label`, `onClick` |
| icon | `ReactNode` | `"+"` | Icon rendered on the FAB trigger |
| label | `string` | `"Speed dial"` | `aria-label` on the group and open menu |

### Accessibility

- Trigger button uses `aria-haspopup="menu"` and `aria-expanded` to reflect the open state.
- When open, the action list renders with `role="menu"` and each action as `role="menuitem"` with a clear `aria-label`.
- Icons on actions are `aria-hidden="true"` so screen readers announce the action label, not the glyph.
- Escape closes the menu and restores focus to the FAB trigger so keyboard users stay oriented.

## ToggleGroup

Grouped toggle buttons — single or multiple selection with full roving-tabindex keyboard support.

<ComponentPreview label="ToggleGroup">
  <ToggleGroupDemo />
</ComponentPreview>

```tsx
import { ToggleGroup, ToggleGroupItem } from "@weiui/react";

// Single — one item active at a time
<ToggleGroup type="single" defaultValue="left">
  <ToggleGroupItem value="left">Left</ToggleGroupItem>
  <ToggleGroupItem value="center">Center</ToggleGroupItem>
  <ToggleGroupItem value="right">Right</ToggleGroupItem>
</ToggleGroup>

// Multiple — any combination active
<ToggleGroup type="multiple" defaultValue={["bold"]}>
  <ToggleGroupItem value="bold">B</ToggleGroupItem>
  <ToggleGroupItem value="italic">I</ToggleGroupItem>
  <ToggleGroupItem value="underline">U</ToggleGroupItem>
</ToggleGroup>
```

### Keyboard
- ArrowLeft / ArrowRight (and ArrowUp / ArrowDown) rove focus between items, wrapping at the ends
- Home / End jump to the first / last item
- Enter / Space toggles the focused item
- Only one item in the group is tabbable at a time (roving tabindex)

### Props

| Prop | Type | Default | Description |
|------|------|---------|-------------|
| type | `"single" \| "multiple"` | `"single"` | One vs many items active at once |
| value | `string \| string[]` | — | Controlled selected value(s) |
| defaultValue | `string \| string[]` | — | Initial uncontrolled value(s) |
| onChange | `(value: string \| string[]) => void` | — | Fires when selection changes |
| disabled | `boolean` | `false` | Disables every item in the group |
| label | `string` | — | Sets `aria-label` on the group wrapper |

### Accessibility

- `type="single"` renders the group as `role="radiogroup"` with each item as `role="radio"` and `aria-checked` kept in sync with the selected value.
- `type="multiple"` uses `role="group"` with each item as `role="button"` carrying `aria-pressed`.
- Roving tabindex — only the focused (or first) item is tabbable, matching the ARIA Authoring Practices pattern for both toggle buttons and radio groups.
- Disabled items are skipped during arrow-key navigation.

## ButtonGroup

Group related buttons with shared borders.

<ComponentPreview label="ButtonGroup">
  <ButtonGroupDemo />
</ComponentPreview>

```tsx
import { ButtonGroup, Button } from "@weiui/react";

<ButtonGroup>
  <Button variant="outline">Copy</Button>
  <Button variant="outline">Cut</Button>
  <Button variant="outline">Paste</Button>
</ButtonGroup>
```

### Props

ButtonGroup forwards every native `<div>` attribute — it adds no WeiUI-specific props. Pair it with any focusable children (typically `Button`), and the CSS handles shared-border collapsing.

### Accessibility

- Root renders as `<div role="group">` so assistive tech announces the buttons as a related set.
- Pair with an `aria-label` when the group's purpose isn't clear from its children (e.g. `aria-label="Clipboard actions"`).
- Unlike `ToggleGroup`, ButtonGroup doesn't manage selection — every child button keeps its own semantics and tab stop.
