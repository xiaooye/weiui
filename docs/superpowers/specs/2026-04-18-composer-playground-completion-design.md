# Composer + Playground Completion — Design Spec

**Date:** 2026-04-18
**Owner:** Wei
**Status:** Draft
**Related work:** Builds on `docs/superpowers/specs/2026-04-18-ai-first-integration-design.md` (registry JSON already ships ~65 components with prop schemas — this spec reuses that).

---

## 1. Goal

Finish the two interactive tool pages (`/playground` and `/composer`) so they actually exercise the full WeiUI library rather than a 5–9 component slice, emit production-ready code (with correct imports including heavy-component subpaths), support nesting, and persist state across reloads. Today both pages are ~15–20% complete: scaffolding is in place but 80% of components render as `<Text>name</Text>` placeholders, generated code has no import statements, and reloading the page resets all state.

**The test for "finished":**
- A user can pick any of 65 components in Playground, tweak every documented prop, copy paste-ready code into a fresh project, and have it compile.
- A user can compose a real multi-component UI (login form: Card → Heading, Field+Input, Field+Input, Button) in Composer, export it, and have a working React file.

---

## 2. Scope

### In scope

**Playground** — interactive prop tweaker:
- All 65 components renderable in preview with live prop updates.
- Type-aware prop controls: string, number, boolean, enum/union, color, object, ReactNode.
- Generated code includes correct imports (main barrel vs subpath for Editor/DataTable/Chart).
- State persistence via URL query params (shareable) + localStorage fallback.
- Toast on copy success.
- Component search/filter + category grouping in the selector.
- Dark/light theme toggle for the preview area (independent of the docs site's theme).
- Responsive preview size presets (mobile / tablet / desktop / full).

**Composer** — drag-to-add multi-component canvas:
- All 65 components renderable in the canvas (not just 9).
- Nested component hierarchy (Card can contain Button + Text + Field).
- Drag-and-drop from palette to canvas; reorder within canvas; drag into containers.
- Schema-aware props editor (enum dropdowns, number inputs, color pickers) powered by the same registry JSON the AI-first work already produced.
- Full-file code export: JSX with imports + a valid standalone HTML snippet with the right CSS imports.
- State persistence (localStorage; URL for share-only flow).
- Keyboard shortcuts: Delete (remove), Ctrl/Cmd+D (duplicate), Ctrl/Cmd+Z / Ctrl/Cmd+Shift+Z (undo/redo).
- Template gallery (5 starter templates: login form, settings page, dashboard card, pricing grid, empty state).
- CodeSandbox / StackBlitz "Open in" export.

**Shared infrastructure:**
- Move to a single `ComponentSchema` data source: read `apps/docs/public/registry/*.json` (already generated by Task 2 of the AI-first spec). No more hand-maintained parallel registries in `/playground/lib/component-registry.ts` and `/composer/lib/component-tree.ts`.
- Unified prop-control component library (`ControlString`, `ControlNumber`, `ControlBool`, `ControlEnum`, `ControlColor`, `ControlObject`, `ControlReactNode`) used by both pages.
- Unified code-generation module that both pages call to produce JSX/HTML output.

### Out of scope

- Visual / Figma-style canvas with arbitrary x/y positioning. Composer stays list-based with nesting; a future spec can upgrade to a visual grid.
- AI-assisted generation ("describe what you want and get a layout"). Future spec.
- Import of Figma frames or existing React code.
- Collaborative editing, comments, version history beyond undo/redo.
- Full theme builder integration in Composer (the `/themes` page already does this; cross-linking is enough).
- Snap guides, rulers, grid overlays on the canvas.

### Data contracts

**`ComponentSchema`** (already exists in `apps/docs/scripts/registry-schema.ts`):

```ts
interface ComponentSchema {
  name: string;
  category: string;
  description: string;
  importPath: string;         // "@weiui/react" or subpath
  subpathImport: string | null;
  dependencies: string[];
  props: PropSchema[];        // { name, type, default?, required?, description }
  compound: string[];
  examples: ExampleSchema[];
  accessibility: string[];
}
```

**New — `ComponentNode`** (for Composer tree):

```ts
interface ComponentNode {
  id: string;                    // stable UUID for React keys + drag handling
  type: string;                  // component name, e.g., "Button"
  props: Record<string, unknown>;
  children: ComponentNode[];     // empty for leaf components
  text?: string;                 // inline text content (e.g., Button label), mutually exclusive with children
}
```

Crucial: this is a **tree**, not a flat array. Containers (`Card`, `Stack`, `Grid`, `Dialog`, `Form`) use `children[]`; leaves (`Button`, `Badge`) use `text?`. Some components (e.g., `Stack`) can hold both — in which case `text` is a label and `children[]` are nested components. Code-gen decides.

**`PlaygroundState`** (for Playground URL/localStorage):

```ts
interface PlaygroundState {
  component: string;             // e.g., "Dialog"
  props: Record<string, unknown>;
  theme: "auto" | "light" | "dark";
  viewport: "mobile" | "tablet" | "desktop" | "full";
}
```

Serialized to URL as `?c=Dialog&p=base64(jsonProps)&t=dark&v=tablet` with a ~4 KB cap (compress prop object via base64).

### Single renderer, two hosts

Both pages share `render-component.tsx`:

```tsx
// Accepts a ComponentNode or a Playground-state slice; renders the real React
// component from @weiui/react (or subpath for heavy ones). Dynamic-import heavy
// components (Editor, DataTable, Chart) to keep initial-load bundle small.
function renderNode(node: ComponentNode): ReactNode;
```

Dynamic imports use Next.js `next/dynamic` with `ssr: false` and a skeleton fallback, preserving the Wave 11 bundle-split wins on tool pages. Main-barrel components render eagerly.

### Code generation

`generate-code.ts` is a pure function from a tree (or single node) to a string:

```ts
interface GenerateCodeOptions {
  target: "jsx" | "tsx" | "html";
  componentWrap?: boolean;        // wrap in export default function
  includeImports?: boolean;
}

function generateCode(root: ComponentNode | ComponentNode[], opts: GenerateCodeOptions): string;
```

Behaviors:
- **JSX/TSX:** emits `import { Button, Dialog } from "@weiui/react";` grouped by path; heavy components go to their subpath (`import { Editor } from "@weiui/react/editor";`). Props serialize correctly (`disabled={true}`, `value={5}`, `label="Email"`). Falsy/default props filtered out. When `componentWrap=true`, wraps in `export default function Composition() { return (...) }`.
- **HTML:** emits a standalone `<!DOCTYPE html>` snippet with CDN links to the CSS bundle (`https://weiui.dev/weiui.min.css`) + appropriate classes (`wui-button wui-button--solid`). Useful for non-React consumers.

Imports come from the registry JSON's `importPath` + `subpathImport` fields — no hardcoded heavy-component list in the code-gen.

## 4. Per-page designs

### 4.1 Playground

**Layout (3-column grid, responsive):**

```
┌──────────────┬────────────────────────────────┬──────────────┐
│              │                                │              │
│   Selector   │         Preview (live)         │    Props     │
│              │                                │              │
│  [Search]    │  ┌──────────────────────────┐  │  variant ▼   │
│  [Category▼] │  │  <Dialog open size="lg"> │  │  [solid ▼]   │
│              │  │    <Button>Save</Button> │  │              │
│  ▸ Form      │  │  </Dialog>               │  │  size ▼      │
│  ▸ Overlay   │  └──────────────────────────┘  │  [md ▼]      │
│    Dialog    │                                │              │
│    Drawer    │  ┌──────────────────────────┐  │  loading     │
│    Popover   │  │ [JSX][HTML] [Copy] [🔗]  │  │  [ ] false   │
│    Tooltip   │  │ import { Dialog } from … │  │              │
│  ▸ Data      │  │ <Dialog open size="lg"> │  │  (search)    │
│              │  │   <Button>Save</Button>  │  │              │
│  [theme◌◐●]  │  └──────────────────────────┘  │              │
│  [📱📱💻]    │                                │              │
└──────────────┴────────────────────────────────┴──────────────┘
```

**Key features:**

1. **Component selector** — collapsible category groups (Form, Overlay, Data, etc.), search box that filters by name + description, keyboard nav (arrow keys + Enter).

2. **Preview area** — iframe-wrapped. Preview sets its own `data-theme` (independent of docs-site theme) and applies a responsive `max-width` based on the viewport preset. Falls back to inline rendering when iframe is impractical (SSR snapshots for screenshot tests).

3. **Props panel** — renders a `PropControl` per prop, chosen by `PropSchema.type`:
   - Union of string literals (`"sm" | "md" | "lg"`) → `ControlEnum` (select/dropdown).
   - `boolean` → `ControlBool` (checkbox/switch).
   - `number` → `ControlNumber` (input + optional slider when min/max known).
   - `string` → `ControlString` (text input; textarea if type mentions `ReactNode`).
   - `ReactNode` (common for `children`) → `ControlReactNode` (text input that wraps in `<>{…}</>`).
   - Color token patterns → `ControlColor` (WeiUI `ColorPicker`).
   - Object → `ControlObject` (JSON textarea with live validation).
   - Unknown → `ControlString` fallback.
   Defaults from `PropSchema.default` are pre-filled.

4. **Code output** — tab between JSX/TSX/HTML. Shiki highlights. Copy button triggers `toast.success("Copied!")`. Share button (🔗) copies a self-contained URL with state serialized.

5. **Theme toggle in preview** — auto/light/dark tri-state button at bottom-left. Uses CSS custom property overrides in the preview iframe.

6. **Viewport presets** — icons for mobile (375px), tablet (768px), desktop (1280px), full. Sets iframe max-width. Active state saved in URL.

**State flow:**

```
URL / localStorage
  ↓
hydrate on mount
  ↓
PlaygroundState (React state)
  ↓ ────────────────────┐
  ↓                     ↓
render <Preview>   serialize to URL (debounced 300ms)
  ↓                     ↓
  ↓               serialize to localStorage (debounced 1s)
  ↓
generate-code → <CodeOutput>
```

URL format: `?c=Dialog&p=eyJ2YXJpYW50IjoibWQifQ==&t=dark&v=tablet`.

### 4.2 Composer

**Layout (3-column grid):**

```
┌──────────────┬────────────────────────────────┬──────────────┐
│              │                                │              │
│   Palette    │         Canvas (nested)        │    Props     │
│              │                                │              │
│  [Search]    │  ▾ Card                        │  variant ▼   │
│  [Template▼] │    ▾ Stack gap=4               │              │
│              │      ─ Heading: "Sign in"      │  size ▼      │
│  ▸ Form      │      ▾ Field                   │              │
│  ▸ Overlay   │        ─ Label: "Email"        │  (select a   │
│    Dialog    │        ─ Input placeholder=…   │   node to    │
│    Drawer    │      ▾ Field                   │   edit)      │
│    Popover   │        ─ Label: "Password"     │              │
│  ▸ Data      │        ─ Input type="password" │              │
│              │      ─ Button: "Continue"      │              │
│  ──────────  │                                │              │
│  Templates   │  [JSX][TSX][HTML] [Copy] [📦] │              │
│  · Login     │  import { Card, Stack, … }    │              │
│  · Settings  │  export default function App() │             │
│  · Dashboard │    return (                   │              │
│              │      <Card>…</Card>            │              │
│  [⌘D ⌫ ↶ ↷]  │    );                          │              │
└──────────────┴────────────────────────────────┴──────────────┘
```

**Key features:**

1. **Palette** — same category structure as Playground's selector. Drag handle on each item (React DnD + HTML5 drag/drop). Click-to-add fallback for accessibility.

2. **Canvas as outline tree** — renders the `ComponentNode` tree as an indented outline. Each row has: disclosure triangle (if container), type label (e.g., `Card`), inline text editor (if leaf with `text`), action icons (duplicate, delete). Drop zones between rows and inside container rows. Selected row highlights; clicking selects.

3. **Drag-and-drop:**
   - Palette → canvas: drag a palette entry into any container node (or root) to insert as a child. Drop indicator animates.
   - Reorder within container: drag a canvas row vertically; sibling rows shift.
   - Reparent: drag a canvas row into a different container.
   - Tree depth cap: 10 levels (prevents runaway nesting).

4. **Preview render under the outline** — below the outline tree, render the actual composition (calls `renderNode()`). Live updates as props change. Toggle to hide if tree is deep.

5. **Props editor** — same schema-driven controls as Playground. Shows props for the selected node; empty state when nothing selected.

6. **Code export (bottom)** — JSX / TSX (full file with `export default`) / HTML / "Open in CodeSandbox" / "Open in StackBlitz". Copy-to-clipboard with toast. Downloadable `.tsx` file for offline use.

7. **Keyboard shortcuts** (registered on `<main>`):
   - `Delete` / `Backspace` — delete selected node.
   - `Ctrl/Cmd + D` — duplicate selected node (and its subtree) as next sibling.
   - `Ctrl/Cmd + Z` — undo.
   - `Ctrl/Cmd + Shift + Z` — redo.
   - `Ctrl/Cmd + C` / `V` — copy/paste subtree.
   - `Escape` — deselect.

8. **Undo/redo** — ring buffer of last 50 tree states. Cheap because nodes are immutable + restructure via React `useReducer`.

9. **Template gallery** — 5 starter templates:
   - **Login form** — Card > Stack (Heading, Field+Input × 2, Button).
   - **Settings page** — Container > Heading + Stack (Field + Input × N, Divider, Button).
   - **Dashboard card** — Card > Stack (Heading, Text, ProgressBar, Stack-row (Chip × 3)).
   - **Pricing grid** — Grid (Card × 3; each Card > Stack: Heading, Text, List of benefits, Button).
   - **Empty state** — EmptyState > (Icon, Heading, Text, Button).
   Each template is a pre-built `ComponentNode` tree serialized as JSON in `templates/`. Click = replace canvas contents (with undo-able history frame).

10. **State persistence** — canvas tree saved to `localStorage` on every change (debounced 500ms). Optionally shareable via `?tree=base64(json)` URL.

11. **CodeSandbox / StackBlitz export** — `generateCode(tree, { componentWrap: true })` wrapped in a minimal `package.json` + `index.tsx` + CSS imports, POSTed to the target service's create-API, returning a URL the user is redirected to.

## 5. Success criteria

The spec is done when:

1. **Coverage** — every one of 65 components is selectable in Playground, renderable in Composer. No `<Text>{name}</Text>` placeholder fallback anywhere.
2. **Prop controls** — for each component, every prop listed in its registry JSON has an appropriate control (string/number/bool/enum/color/object/ReactNode). No raw text input as the sole fallback for non-string props.
3. **Code generation correctness** — copied code compiles in a fresh project:
   - All needed imports present, grouped by module, subpath-aware (`@weiui/react/editor`, etc.).
   - Boolean props serialize as `disabled={true}` not bare `disabled` when explicit.
   - String literal unions quote correctly.
   - Nested children emit proper indentation (2 spaces per level).
4. **Nesting** — Composer canvas supports trees ≥ 5 levels deep. Drag into a container works; reorder works; escape-from-container works.
5. **State persistence** — refreshing Playground or Composer restores the last edited state. Sharing a URL reproduces the same state on another machine.
6. **Keyboard shortcuts** — all 6 Composer shortcuts work; Playground has `Ctrl/Cmd + C` on the code output + `Escape` to blur inputs.
7. **Toasts** — copy-to-clipboard shows `toast.success`; failures show `toast.error`.
8. **Templates** — 5 templates load correctly in Composer; undo works after template load.
9. **CodeSandbox/StackBlitz export** — clicking "Open in …" opens a new tab with the compositor tree running.
10. **Performance** — Playground state update → preview paint < 80 ms. Composer tree update (add/move/delete) < 50 ms up to 200 nodes.
11. **Tests** — unit tests for tree reducer, code-gen (including subpath imports, props serialization, nesting), state serialization. Playwright e2e: select-Dialog-tweak-props-copy-code (Playground); drag-three-components-export-code (Composer).
12. **Docs build** — `pnpm --filter @weiui/docs build` produces clean HTML for both pages (no hydration errors).

## 6. Non-goals

- AI-assisted generation ("describe a login form; generate tree"). Deferred; would rely on MCP server tools (`get_component`, `get_example`) from the AI-first pass.
- Figma plugin or Figma-to-tree import.
- Visual spatial canvas (arbitrary x/y, resizable boxes). Outline tree is sufficient for v1.
- Collaborative editing / multi-user.
- Version history beyond undo/redo.
- Theming customization from inside the tools (use `/themes` page).
- Runtime prop type inference from TypeScript — stick with the static registry JSON (already ts-morph–derived). If a prop's `type` string is too ambiguous for a `PropControl` mapping (e.g., `ReactNode | string | (() => ReactNode)`), fall back to a text input; document this limitation.

## 7. Risks & mitigations

| Risk | Mitigation |
|------|------------|
| Rendering all 65 components raises bundle size (the Composer canvas imports them all) | Use `next/dynamic({ ssr: false })` for heavy components; code-split by category. Cap initial tool-page First-Load JS at ≤ 200 kB, enforced by CI check (reuse the bundle-size check from Polish 11). |
| Drag-and-drop library bloat | Use native HTML5 DnD via a thin wrapper (~1 KB) instead of `react-dnd`. Only reach for `react-dnd` if nested-drop logic requires it. |
| State serialization hits URL length limits | Cap `?p=` (props) at 4 KB; beyond that, fall back to localStorage-only and show a warning. Tree serialization limit at 8 KB; beyond, switch to localStorage. |
| CodeSandbox/StackBlitz API breakage | Fall back to downloadable `.zip` bundle if POST fails; log telemetry. |
| Registry JSON drifts from implementation (Prop exists in code but not in JSON, or vice versa) | Already mitigated by AI-first Task 2 — generator parses the actual TS AST. Add a CI check that runs `pnpm --filter @weiui/docs build` + verifies registry JSON contains an entry per `packages/react/src/components/*` folder. |
| Prop-control library bloat | The 7 controls are tiny (~50 LOC each). Share with docs `/docs/tokens` page or any other doc pages needing prop inputs. |
| Nesting creates invalid compositions (e.g., `<Button>` containing `<Dialog>`) | Add an optional `allowedParents` / `allowedChildren` field to `ComponentSchema` (future; not v1). For v1: allow any nesting; if invalid, user sees a runtime error in preview and can fix. |

## 8. Implementation plan

Plan file: `docs/superpowers/plans/2026-04-18-composer-playground-completion.md`.

Eight tasks, each shipping incrementally usable output:

1. **Shared schema loader + prop-controls library.** `component-schema-loader.ts` + 7 `ControlX` components.
2. **Playground: 65-component renderer + generate-code with imports.** Replaces the 5-case switch; code output now compiles.
3. **Playground: schema-driven PropsPanel + URL/localStorage state + copy toast.** Every prop editable; shareable URLs.
4. **Playground: theme + viewport toggles + search/category grouping.** Polish features.
5. **Composer: nested `ComponentNode` tree + outline canvas + live preview.** Replaces the flat list.
6. **Composer: drag-and-drop + keyboard shortcuts + undo/redo.** Core editing UX.
7. **Composer: schema-driven PropsEditor + full-file code export + template gallery.** Production-ready export.
8. **Composer: CodeSandbox / StackBlitz integration + final verification + tests.**

Each task ends with `pnpm build && pnpm test && pnpm --filter @weiui/docs build` green. The plan file spells out bite-sized TDD steps per task.
